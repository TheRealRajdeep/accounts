# Custom Adapter

The Tempo Accounts SDK exposes a low-level `Adapter.define` API for authoring custom adapters. Use it when none of the built-in adapters ([`tempoWallet`](/docs/adapters/tempo-wallet), [`webAuthn`](/docs/adapters/webauthn), [`turnkey`](/docs/adapters/turnkey), [`privy`](/docs/adapters/privy), [`secp256k1`](/docs/adapters/private-key)) match your custody model.

Use it when you want:

- A signer not covered by a built-in adapter.
- Full control over the JSON-RPC actions exposed to the provider.
- A thin wrapper that adds metadata or middleware around an existing adapter.

:::tip
If you use Turnkey or Privy, start with their built-in adapters before writing a custom adapter. They already cover session restore, account loading, provider signing, and the callback shape for app-owned auth.
:::

:::tip
If your signer can produce a private key or sign arbitrary digests, consider wrapping the [`local`](/docs/api/local) adapter first — it implements every required action for you and only asks you to supply `createAccount` and `loadAccounts`. Reach for `Adapter.define` directly when you need to override signing, request translation, or session lifecycle behavior.
:::

## Anatomy

An adapter is a function annotated with display metadata. `Adapter.define(meta, setup)` returns that function, the provider calls it once at startup, and the returned `Instance` wires actions into the provider's `request()` dispatcher.

```ts
import { Adapter } from 'accounts'

const myAdapter = Adapter.define(
  {
    name: 'My Wallet',
    rdns: 'com.example.mywallet',
    icon: 'data:image/svg+xml;base64,...',
  },
  ({ getAccount, getClient, storage, store }) => {
    return {
      actions: {
        async createAccount(params, request) {
          // ...
        },
        async loadAccounts(params, request) {
          // ...
        },
        async signPersonalMessage({ address, data }) {
          // ...
        },
        async signTypedData({ address, data }) {
          // ...
        },
        async signTransaction(params, request) {
          // ...
        },
        async sendTransaction(params, request) {
          // ...
        },
        async sendTransactionSync(params, request) {
          // ...
        },
      },
      cleanup() {
        // ...
      },
    }
  },
)
```

## Error mapping

Wrap signer-specific errors in [`ox`](https://oxlib.sh)'s `Provider` errors so app code can handle them without knowing your signer's internals.

```ts twoslash
import { Provider as ox_Provider } from 'ox'

declare function callMySigner(): Promise<unknown>

try {
  await callMySigner()
} catch (error) {
  // Translate auth failures into the provider contract.
  throw new ox_Provider.UserRejectedRequestError(error as Error)
}
```

Common mappings:

- User-cancelled prompts → `ox_Provider.UserRejectedRequestError`.
- Missing or expired session → `ox_Provider.UnauthorizedError`.
- Methods you choose not to implement → `ox_Provider.UnsupportedMethodError`.
- Malformed adapter input → `ox_Provider.InvalidParamsError`.
